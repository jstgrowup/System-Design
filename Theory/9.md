## 📝 Detailed Notes

### What is Caching? Core Meaning and Benefits

- Caching involves storing frequently accessed data that doesn't change every second in a temporary storage area.
- **Core Objectives:**
  - **Decrease Latency:** Makes the system significantly faster.
  - **Reduce Database Load:** Prevents expensive SQL queries from running repeatedly.
  - **Reduce Cost:** Indirectly saves money by lowering system resource consumption.
- 💡 **Analogy:** A home page showing "5000+ Students" or "100+ Projects." If this data only updates every 6 hours, running a complex SQL query for every one of the 5,000 daily visitors is wasteful. Caching the result once for 6 hours saves 4,999 redundant queries.

### The Logic of Caching: Cache Hit vs. Cache Miss

- **Cache Hit:** When the requested data is found in the cache. The system responds immediately.
- **Cache Miss:** When data is not in the cache. The system must perform a "DB Call," fetch the data, save it to the cache for next time, and then respond.
- 🎯 **Interview Tip:** System designers aim to maximize the "Hit Ratio" to keep the database load low and performance high.

### Time To Live (TTL) and Data Freshness

- **TTL:** A fixed amount of time for which a cache entry remains "alive" or valid.
- Once the TTL expires (e.g., 6 hours), the cache is automatically deleted.
- The next request will result in a **Cache Miss**, forcing a refresh of the data from the source.
- 💡 **Example:** LinkedIn might use a 2-minute TTL for post likes. Users might see slightly outdated like counts for 120 seconds, but it saves millions of database queries.

### Types of Caching

- **Client-Side:** Storing data in the user's browser (Local Storage) or mobile app.
  - ⚠️ **Warning:** This can lead to "Stale Data." You might see one like count on your mobile app (cached) and a different one on your laptop (fresh).
- **Application-Side:** Using tools like **Redis** or **Memcached** within the backend.
- **Database Caching:** Saving large SQL query responses within the DB layer.
- **CDN Caching:** Caching static files like images and videos on servers closer to the user.

### Introduction to Redis: In-Memory Storage

- **In-Memory:** Redis stores data in **RAM**, not on a physical disk. This makes it extremely fast (microsecond response times).
- **Key-Value Store:** Data is stored as simple pairs (e.g., Key: "Homepage_Data", Value: "[JSON Data]").
- **External vs. Internal:** While a main SQL database is an external service that takes time to query, Redis acts as a "near-server" storage that is much faster to access.

### Redis Data Structures

- **Strings:** Standard text (e.g., user profiles).
- **Hashes:** Storing objects with multiple fields.
- **Lists:** Used for building queues.
- **Sets:** Storing unique elements (no duplicates).
- **Sorted Sets:** Elements stored with a "score" for ordering.
  - 🎯 **Interview Tip:** Use Sorted Sets to build **Leaderboards** for millions of users efficiently.

### Cache Eviction Policies

When the cache (RAM) is full, the system must decide what to delete:

- **LRU (Least Recently Used):** Deletes the data that hasn't been accessed for the longest time. This is the most common policy.
- **LFU (Least Frequently Used):** Deletes data with the lowest access count.
- **FIFO (First In First Out):** Deletes the oldest data first, regardless of how often it's used.

### Content Delivery Networks (CDN)

- **Edge Servers:** CDN places "static content" (images, JS, CSS, videos) on servers geographically closer to the user.
- 💡 **Analogy:** If your main server is in the USA but your user is in India, downloading a 10MB image from the USA might take 10 seconds due to distance and "environmental factors." A CDN edge server in Mumbai can deliver that same image in 1 second.
- **Providers:** **Cloudflare** (popular for its free tier and SSL), **Akamai**, and **AWS CloudFront**.

### Advanced Challenges: Cache Invalidation

- **Stale Data:** When the cache shows old information because the source data changed before the TTL expired.
- **Hard Problem:** "Cache invalidation is hard" is a famous saying. It's difficult to perfectly sync cache and DB.
- **Solutions:**
  - **Manual Purge:** Manually deleting the cache when an update occurs.
  - **Versioning:** Using URLs like `image_v2.png` to force the CDN to fetch the new version.
  - **Write-through/Write-back strategies**.

### 1. The Caching Logic Flow

```text
[Client Request] --> [Check Cache (Key)]
                          |
        ------------------------------------------
        | (Hit: Found)                | (Miss: Not Found)
        V                             V
[Return Data Fast]          [Query SQL Database]
                                      |
                            [Save Result to Cache]
                                      |
                             [Return Data to User]
```

_Description: Demonstrates the "If-Else" logic used to decide between a fast cache response and a slow DB query._

### 2. CDN Proximity Logic

```text
[Original Server: USA] ----(10,000 miles)----> [User: India] = 10s Latency
           |
           V (Copy Static Files)
           |
[Edge Server: Mumbai] ---(200 miles)--------> [User: India] = 1s Latency
```

_Description: Shows how CDNs reduce latency by minimizing the physical distance data travels._

// Pseudo-code logic for checking a cache before a DB call
if (cache.exists(key)) {
  // Cache Hit
  return cache.get(key);
} else {
  // Cache Miss
  const data = db.query("SELECT * FROM table...");
  cache.set(key, data, ttl); // Save for next time
  return data;
}

## 🔁 Comparisons & Trade-offs

| Feature       | Redis (Application Cache)         | CDN (Content Delivery Network)        |
| :------------ | :-------------------------------- | :------------------------------------ |
| **Data Type** | **Dynamic** (JSON, SQL results)   | **Static** (Images, Videos, JS)       |

## 🎯 Interview Questions

1. **Why is Redis faster than a traditional SQL database?**
   Because Redis is an "In-Memory" store that keeps data in RAM, avoiding slow disk I/O operations.
2. **Explain the concept of TTL and why it's used.**
   Time To Live defines how long data stays in the cache. It ensures the data doesn't become too "stale" by forcing a refresh after a set period.
3. **What happens during a "Cache Miss"?**
   The application must fetch data from the primary database, which is slower, and then update the cache for future requests.
4. **When should you NOT use caching?**
   When data changes every second or is highly unique to every single request, making the "Hit Ratio" near zero.
5. **How does a CDN improve user experience?**
   By serving heavy static files from an "Edge Server" physically closer to the user, it significantly reduces download time.

## 📌 Key Takeaways

- Use **Redis** for dynamic data (DB results) and **CDNs** for static data (media).
- **TTL** is a trade-off between performance and data freshness.
- **LRU** is the standard strategy for managing full caches.
- Real-world giants like **Netflix** use Redis for metadata (descriptions) and CDNs for the actual video files.
- Always prioritize **Cache Hits** to protect your database during traffic spikes.
