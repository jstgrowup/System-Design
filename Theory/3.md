### Defining System Design & HLD

- **System Design Overview:** Focuses on scalability, reliability, security, and "reverse engineering" (intentionally trying to fail the system to fix it).
- **High-Level Design (HLD):**
  - Focuses on the **overall system architecture**.
  - Describes how **major components** work together.
  - Includes system overview, tech stack, data flow, and scalability options.
  - Involves external integrations like **Redis** (caching) or **RabbitMQ** (queuing).

### Core Components of HLD

- **Load Balancer:** Distributes traffic.
- **API Gateway:** Entry point for technical requests.
- **Microservices:** The structure of various services.
- **Database (DB) Design:** High-level choice of storage.
- **CDN (Content Delivery Network):** For content delivery.
- **Caching & Storage:** Using tools like **S3 Buckets** for images.
- 🎯 **Interview Tip:** Using technical terms like "Load Balancer" and "API Gateway" when asked about HLD makes your response more effective.

### Case Study: URL Shortener Functionality

- **Concept:** Taking a long, complex URL (e.g., a Google Drive link) and converting it into a short one (e.g., `tinyurl.com/drive1pdf`) that redirects back to the original.
- **Functional Requirements:**
  1. Shorten long URLs.
  2. Redirect short URLs to long URLs.
  3. Track clicks (e.g., if 500 people clicked).
- **Non-Functional Requirements:**
  1. **Scalability:** Must handle surges (e.g., 1M+ clicks if a celebrity like Carry Minati shares the link).
  2. **Low Latency:** Redirection must be fast, not take minutes.
  3. **Availability:** The system must stay online.

### High-Level Components of a URL Shortener

- **Web Server:** To receive requests.
- **Hash Generator:** To create a random 8-digit unique ID for the short URL if a custom one isn't provided.
- **Database:** Stores IDs, Long URLs, Short URLs, and click counts.
- **Caching Logic:** Save DB operations by caching URLs that get high traffic (e.g., 300+ or 3000+ clicks in a minute).

### Defining Low-Level Design (LLD)

- **Definition:** The **detailed implementation** stage involving specific classes, methods, and their interactions.
- 💡 **Analogy:** If building a house, HLD is the land, the number of floors, the pillars, and the walls. LLD is the internal wiring, plumbing, painting, and detailed connections.
- **LLD Components:**
  - Class diagrams.
  - Detailed Database Schema (SQL vs. NoSQL selection).
  - API request/response contracts (data formats).
  - Object relationships and flow diagrams.
  - **Error Handling & Edge Cases:** Crucial for robust LLD.

### URL Shortener LLD: DB & Classes

- **DB Table Structure:** Columns include `ID` (Primary Key), `Long URL` (String), `Short URL` (String/Tail), `Clicks` (Long Int), and `Created At` (Timestamp).
- **URL Service Class:** Includes methods like `createURL()`, `generateHash()`, and `redirect()`.
- **Instagram Comparison (LLD):** Designing separate tables for Users (ID, name, bio, profile pic), Posts (ID, caption, image URL, User ID as foreign key), and Comments, and defining how they link.

### API Design for URL Shortener

- **Endpoint:** `POST /shorten`
- **Request Body:** Contains `longURL` and an optional `tail` (custom name).
- **Logic Flow:**
  1. Check if the requested `tail` already exists in the DB.
  2. If it exists, return an error code ("Tail already used").
  3. If it doesn't exist (or no tail was provided), use the **Hash Generation Service** to create one.
  4. Store the mapping and return the short URL as the response.

## 🎯 Interview Questions

1. **What is the goal of "Reverse Engineering" in system design?**
   It involves intentionally trying to fail the system to identify weak points and then fixing them to improve reliability.
2. **What are the common components discussed in HLD?**
   Load Balancer, API Gateway, Microservices, Databases, CDNs, and Caching tools like Redis.
3. **Difference between functional and non-functional requirements?**
   Functional requirements are the "what" (e.g., shortens URL), while non-functional requirements are the "how well" (e.g., latency, scalability).
4. **How would you optimize a high-traffic URL redirection?**
   By implementing caching (like Redis) for URLs that exceed a specific click threshold (e.g., 3000 clicks/min) to avoid repeated DB hits.
5. **Where does DB schema design sit in the HLD/LLD split?**
   While the choice of DB (SQL vs. NoSQL) is HLD, the detailed schema, table names, and specific attributes are part of LLD.

## 📌 Key Takeaways

- HLD provides the **broad overview**; LLD provides the **detailed logic**.
- Always start with **Requirement Gathering** (Functional and Non-functional).
- **Caching** is a critical HLD decision to manage high Requests Per Second (RPS).
- **LLD** involves mapping out class relationships and specific API data contracts.
- Real-world design includes handling **edge cases**, like tail collisions in URL shorteners.
