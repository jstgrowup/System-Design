## 📝 Detailed Notes

### What is Indexing? (Definition & Analogy)

- **Definition:** A technique used to speed up data retrieval in databases.
- 💡 **Analogy:** Like a thick dictionary or a book index. Instead of reading the entire book to find a chapter (Full Scan), you check the index for the page number and jump directly to it.
- **Performance Impact:**
  - **Without Index:** Database performs a full scan. For 10 million records, the time complexity is **$O(N)$**.
  - **With Index:** Retrieval time drops to **$O(\log N)$**, significantly decreasing search time.

### The Mechanics of Indexing

- **Primary Structure:** Most databases use **B-Trees** for indexing. Other types include Hash Trees and Inverted Indexes (specifically for search engines).
- **Logic:** It works on a **"Divide and Conquer"** principle similar to binary search.
- **B-Tree Visualization:** If searching for 145, the system checks if it lies between 100 and 155. It disregards all other value ranges and follows the pointer to the subset containing values between 100 and 155, repeating this until the specific record is found.

### Composite Indexing & Syntax

- **Composite Index:** Indexing multiple columns simultaneously (e.g., Name and City) for queries that use multiple filters.
- ⚠️ **Warning:** In composite indexing, **order matters**. If you create an index on `(Name, City)`, searching by `City` first may not utilize the index correctly.

### Real-World Indexing Examples

- **Instagram:** Used for instant "Username Not Available" warnings when creating a profile.
- **Amazon:** Used to check if a Product ID already exists in the catalog.

### Indexing Trade-offs: When NOT to Use It

- **Small Tables:** If a table only has 100 records, a full scan is fast enough. Using a heavy "axe" (indexing) to cut a thread is unnecessary.
- **High Write Systems:** Indexing **slows down write operations**. Every time new data is added, the index must also be updated, consuming performance.
- **Memory Overload:** Indexes consume significant storage space, often reaching **GBs** in size. Avoid if you have strict memory constraints.

### What is Sharding? (Horizontal Scaling)

- **Definition:** Partitioning a massive database into smaller, manageable pieces ("shards") across multiple servers.
- **Goal:** To achieve **Horizontal Scaling**. For example, a 100GB database can be split into two 50GB databases on different machines.

### Sharding Strategies

- **Range-based (Directory-based):** Splitting data by value ranges. For example, a dictionary split into two books: A–M in Dictionary 1 and M–Z in Dictionary 2.
- **Hash-based:** Using a mathematical logic (e.g., `ID % 4`) to distribute data across four different shards.
- **Directory-based:** Mapping specific keys directly to a specific shard/dictionary.

### Architectural Tiers: Horizontal vs. Vertical Partitioning

- **Horizontal Partitioning:** Splitting by rows. Example: Moving users with names A–M to Database 1 and M–Z to Database 2.
- **Vertical Partitioning:** Splitting by columns or finer subsets. Example: If names starting with "A" are too many, further splitting the "A" group into A1 (A–Am) and A2 (Am–Az).

### Sharding Challenges

- **No Joins:** It is extremely difficult to perform "Joins" between tables sitting on different shards.
- **Complexity:** Managing distributed transactions and rebalancing shards is very difficult.
- **Hot Partitions:** Some shards may receive much higher traffic than others (e.g., many users have names starting with "A" but few with "Z").

### What is Replication? (Data Safety & Load Distribution)

- **Definition:** Copying the same data onto multiple servers.
- **Primary Goals:**
  1. **Fault Tolerance:** If one data center burns down or loses power, data is safe on a second server.
  2. **Read Scaling:** Multiple users can read from different copies of the database simultaneously, reducing the load on a single machine.

### Replication Types

- **Master-Slave (Primary-Replica):**
  - **Master:** Handles all **Write** operations.
  - **Slave:** Handles all **Read** operations.
- **Multi-Master:** Multiple nodes can handle writes and must sync with each other.
- **Leaderless:** Every node handles its own reads/writes and syncs (e.g., Amazon Dynamo).

### Consistency Lag: The LinkedIn Example

- 💡 **Example:** You might see a different number of likes for a LinkedIn post on your phone vs. your laptop. This happens because the **Write** occurred in the Master, but you are **Reading** from a Slave that hasn't finished copying the data yet (Lag). This is known as being **eventually consistent**.

## 🔁 Comparisons & Trade-offs

| Feature          | Indexing                  | Sharding                | Replication       |
| :--------------- | :------------------------ | :---------------------- | :---------------- |
| **Primary Goal** | Fast Read/Lookup          | Horizontal Scaling      | High Availability |
| **Pros**         | Speed, Optimized Reads    | Storage/Write Scaling   | Fault Tolerance   |
| **Complexity**   | Low                       | Very High               | Medium            |
| **Trade-off**    | Slows Writes; Uses Memory | No Joins; Complex Logic | Stale Data (Lag)  |

## 🎯 Interview Questions

1. **What is the time complexity of a search with and without an index?**
   Without an index, it is $O(N)$ (Full Scan); with an index (B-Tree), it is $O(\log N)$.
2. **Why does indexing slow down write operations?**
   Because every time a record is written, the database must also update the index pointers to ensure future searches are accurate.
3. **What is the difference between Master-Slave and Multi-Master replication?**
   In Master-Slave, only one node handles writes; in Multi-Master, multiple nodes can handle writes and must synchronize.
4. **When should you avoid sharding?**
   Avoid sharding if your system requires complex joins across tables or if the overhead of managing distributed transactions outweighs the scaling benefits.
5. **How does sharding enable horizontal scaling?**
   By breaking a large database into smaller shards, you can place them on separate, cheaper machines instead of one massive, expensive server.

## 📌 Key Takeaways

- **Indexing** speeds up reads but consumes memory and slows writes.
- **Sharding** partitions data to scale horizontally but breaks joins.
- **Replication** provides fault tolerance and read-scaling but can lead to data lag.
- Use **B-Trees** for standard lookups and **Inverted Indexes** for search engines.
- A mature system uses all three: **Index** for lookups, **Shard** for massive scale, and **Replicate** for 24/7 availability.
