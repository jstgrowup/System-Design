## 📝 Detailed Notes

### Foundations of Client-Server Architecture

- **Definition:** A computing model where a **Client** requests services and a **Server** provides them over a network.
- **The Client:** The device or user-facing interface (front-end) that prepares and sends requests.
- **The Server:** The machine or software (back-end) that processes logic, runs queries, and generates responses.

### Domains, Subdomains, and IP Addresses

- **IP Addresses:** Every machine on the network has a unique numerical identity (e.g., 1.1.1.1). Because these are hard to remember, we use **Domain Names**.
- **Domains & Subdomains:**
  - **Domain:** `google.com` or `facebook.com`.
  - **Subdomain:** `mail.google.com` or `blog.google.com`. Each can point to a different server instance or "node".
- **The Directory:** When you type a URL, the request hits an internet directory to find the linked IP address and locate the specific server.

### The Three-Tier Process

1.  **Client (Front-end):** Where the request is made (Website, App, Smart Watch).
2.  **Server (Back-end):** Processes the logic (e.g., Node.js instances). It prepares the response based on the request type (e.g., `/home` or `/profile`).
3.  **Database (DB):** Stores data (e.g., Instagram likes, chat history). The server pulls data from the DB to send back to the client.

- 💡 **Analogy:** LinkedIn works identically on your phone and laptop because they both hit the same centralized database.

### Network Protocols & Connectivity

- **The Medium:** Communication happens over a network consisting of towers, 4G/5G, and intermediate "hops".
- **Protocols:** Rules that ensure safe communication between different devices.
  - **HTTP/HTTPS:** Hypertext Transfer Protocol (Secure).
  - **TCP/IP:** Fundamental networking rules.
  - **WebSockets:** For real-time communication.
- 💡 **Analogy:** Protocols are like **traffic lights**—everyone must follow them globally to avoid chaos and ensure data reaches safely.

### Case Study: Instagram Login Flow

- **Step 1:** User enters username and password on the login page.
- **Step 2:** Request hits the `/login` API endpoint on the server.
- **Step 3 (The Match):** The server logic searches the DB for that specific username. It compares the stored password with the one sent in the request.
- **Step 4 (Response):**
  - If they match (e.g., `abc == abc`), the server returns the user's feed, profile, and chats.
  - If they don't match, the server throws an "Incorrect Password" error.

### Architectural Tiers

- **One-Tier Architecture:** Client, Server, and DB are in one system. Used for simple apps like calculators or To-Do lists.
- **Two-Tier Architecture:** Direct communication between client and server/DB.
- **N-Tier (Multi-Tier) / Distributed:** Involves multiple layers including Load Balancers, Caching layers, and Queues.

### Monolithic vs. Microservices

- **Monolithic:** Everything (Front-end, Back-end, DB) is in a single, large codebase. Harder to scale and a single fault can bring down the entire system.
- **Microservices:** The system is divided into specialized servers (e.g., one for Payments, one for User Management, one for Courses).
  - **Communication:** Services interact via APIs.
  - **Benefit:** Highly scalable and secure. If the Payment service fails, the User Management service remains safe.
  - **Trade-off:** Management is more complex and requires more effort.

### Peer-to-Peer (P2P) vs. Client-Server

- **Client-Server:** Centralized. Easy to update data in one DB, but vulnerable to hacking or central failure.
- **Peer-to-Peer:** Decentralized. Every "node" has a copy of the data (e.g., a series 1-2-3-4 stored across 1000 nodes).
  - **Scalability:** High coordination between nodes.
  - **Security:** Extremely hard to change or corrupt because you would have to change it across millions of nodes.
  - **Examples:** Blockchains, Torrents.

```javascript
// Logic for handling a login request on the server
app.post("/login", (req, res) => {
  const { username, password } = req.body;

  // Server logic: Query DB for user details
  const user = db.findUser(username);

  if (user.password === password) {
    // Success: Return user data in JSON format
    return res.json({
      status: "success",
      feed: user.feed,
      profile: user.profile,
    });
  } else {
    // Error: Return specific error message
    return res.status(401).send("Incorrect Password");
  }
});
```

## 🔁 Comparisons & Trade-offs

| Feature        | Client-Server Architecture         | Peer-to-Peer (P2P)                          |
| :------------- | :--------------------------------- | :------------------------------------------ |
| **Control**    | Centralized (Server controls data) | Decentralized (Every node is equal)         |
| **Data Sync**  | Easy to sync via central DB        | Strong coordination needed; data replicated |
| **Security**   | Vulnerable to single hacking point | Extremely secure (Blockchain/Torrents)      |
| **Management** | Easier management/updates          | Very hard to change data globally           |

## 🎯 Interview Questions

1.  **What is the "single point of failure" in a client-server model?**
    If the central database or the primary backend server fails, the entire system can become unavailable.
2.  **Explain the difference between a domain and an IP address.**
    An IP address is the numerical identity of a machine (1.1.1.1), while a domain is a human-readable name (google.com) that points to that IP.
3.  **Why use Microservices over Monolithic architecture for large systems?**
    Microservices allow for independent scaling, better fault isolation (one service failing doesn't kill the whole system), and more secure management of specific data.
4.  **What is the role of a Load Balancer in a distributed system?**
    It divides the incoming request load across multiple backend servers to ensure no single server is overwhelmed.
5.  **How do you handle "Robot" or "DDoS" attacks in system design?**
    By implementing security measures like Firewalls, CAPTCHAs, and limiting the number of requests a single IP can make.

## 📌 Key Takeaways

- The client-server model is the backbone of the modern internet (Web, Mobile, IoT).
- **Microservices** offer better scalability and fault tolerance than **Monolithic** architectures.
- Security must be baked in at every layer (Firewalls, CAPTCHAs) to protect the DB from corruption.
- **Client-Server** is preferred for centralized apps (Banking/Instagram), while **P2P** is used for decentralized systems (Blockchain).
- **Performance** depends on network protocols and efficient request-response cycles.
