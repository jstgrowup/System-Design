## 📝 Detailed Notes

### HTTP/HTTPS: The Internet’s Protocols

- **HTTP (Hypertext Transfer Protocol):** A communication protocol that facilitates interaction between a Client (browser/app) and a Server.
- **HTTPS:** The "S" stands for Secure; it is the most popular protocol used globally today.
- 💡 **Analogy:** Protocols are like **traffic lights** (Red/Green) or city rules. Without them, there would be chaos and accidents; with them, communication is safe and predictable.

### The Anatomy of an HTTP Request & Response

- **Request Structure:** Contains a URL, a Method (GET/POST), Host info, Headers (Authorization like Bearer tokens, Content-Type), and sometimes a Body.
- **Response Structure:** Includes a Protocol version, a Status Code (e.g., 200 OK), Headers, and the actual Data (Body) usually displayed on the front-end.
- Data in the Body is often hashed or encrypted for security.

### CRUD Operations & HTTP Methods

- **CRUD** stands for Create, Read, Update, and Delete.
- **POST:** Used for **Create** (e.g., creating a Facebook profile or writing a comment).
- **GET:** Used for **Read** (e.g., viewing a profile or post).
- **PUT/PATCH:** Used for **Update**. **PUT** typically performs a full data update, while **PATCH** allows for partial updates (e.g., just changing a phone number).
- **DELETE:** Used for **Delete**.

### HTTP Status Codes

- 🎯 **Interview Tip:** Understanding status code prefixes is vital for debugging.
- **2xx (Success):** "Green Light"—everything is okay (e.g., 200 OK, 201 Created).
- **4xx (Client Error):** "Red Light"—the error is on your side (e.g., 400 Bad Request, 401 Unauthorized, 404 Not Found).
- **5xx (Server Error):** The error is on the server's side (e.g., 500 Internal Server Error).

### Characteristics of HTTP

- **Stateless:** The server does not remember the client. Each request is independent and must contain all necessary info (like auth tokens) to be processed.
- 💡 **Analogy:** Unlike WhatsApp, which requires a "stateful" pipeline for message tunneling, HTTP doesn't care who is asking as long as the request is valid.
- **Text-based:** Human-readable and works over **TCP**.

### REST API: Principles and Architectural Style

- **REST (Representational State Transfer):** An architectural style that uses HTTP/HTTPS in a standardized way.
- **Principles:** Separation of Client and Server, Statelessness, Uniform Interface, and **Resource-based URLs**.
- **Design Advantage:** Instead of creating multiple complex endpoints like `/updateUser`, you use the same resource `/users/id` and simply change the HTTP method (GET, POST, PUT) to perform different actions.

### RPC (Remote Procedure Call) & gRPC

- **RPC:** Allows a service to call a function on another service as if it were a local function call.
- 💡 **Analogy:** Like a `Calculator` class in code—you just call `calc.add(4, 5)` and it returns `9`. RPC does this across different machines over the network.
- **gRPC:** Google’s high-performance framework that uses **HTTP/2** and **Protocol Buffers (Binary)** instead of JSON, making it much faster.
- **Action-based:** Unlike REST (resource-based), RPC is based on executing specific actions.

## 📊 Diagrams & Visual Explanations

### 1. HTTP Protocol "Traffic Light" Logic

```text
[ 2xx ] --> Green Light --> Success (Proceed)
[ 4xx ] --> Red Light   --> Client Error (Fix your request)
[ 5xx ] --> Red Light   --> Server Error (Wait for fix)
```

_Description: Illustrates how developers use status codes to determine the health of a request._

### 2. REST (Resource-based) vs. Non-REST (Action-based)

```text
NON-REST (Verb-heavy)              REST (Resource-centric)
/getUser?id=1                      GET /users/1
/updateUser?id=1                   PUT /users/1
/deleteUser?id=1                   DELETE /users/1
```

_Description: REST uses the HTTP Method to define the action on a static resource URL._

### 3. RPC Call Flow

```text
[ Client Service ] --(Function Call: getUser(123))--> [ Network ]
                                                         |
[ Response: UserData ] <--(Returns Result)----------- [ Server Service ]
```

_Description: RPC makes network calls look like local code execution._

## 🔁 Comparisons & Trade-offs

| Feature             | REST                       | RPC (gRPC)                          |
| :------------------ | :------------------------- | :---------------------------------- |
| **Philosophy**      | Resource-based (`/users`)  | Action-based (`getUser()`)          |
| **Data Format**     | JSON/XML (Human-readable)  | Binary/Protobuf (Machine-optimized) |
| **Performance**     | Medium (Text-heavy)        | High (Fast binary)                  |
| **Use Case**        | Public APIs, Web/Mobile FE | Internal Microservices (M2M)        |
| **Browser Support** | Excellent                  | Limited (Not browser-friendly)      |

## 🎯 Interview Questions

1. **What is the difference between PUT and PATCH?**
   PUT is used for a full update (replacing the resource), whereas PATCH is for partial updates (e.g., updating just one field).
2. **Why is HTTP called "stateless"?**
   Because the server does not store any information about the client session; every request must be self-contained.
3. **What are the common issues with REST APIs?**
   Over-fetching (getting more data than needed) and Under-fetching (getting less data than needed), and being slower than binary protocols.
4. **When would you use gRPC over REST?**
   For internal microservice communication where low latency and high performance are required, such as in banking or stock market systems.
5. **How does RPC handle service communication?**
   It allows one machine to call a function on another machine as if it were a local function in its own code.

## 🔗 Connections to Other System Design Topics

- **Microservices:** RPC/gRPC is the preferred method for internal communication between services.
- **Scalability:** REST’s statelessness makes it easier to scale horizontally since any server can handle any request.
- **Low Latency:** gRPC is a key tool for achieving the low latency requirements discussed in Episode 05.

## 📌 Key Takeaways

- **HTTP** is the fundamental protocol; **REST** is an architectural style built on top of it.
- **REST** is best for public-facing web and mobile APIs because it is simple and human-readable.
- **RPC (gRPC)** is ideal for internal backend-to-backend communication due to its high speed and efficiency.
- **Status Codes** are the "language" of the web—2xx is success, 4xx is your fault, 5xx is the server's fault.
- Real-world systems like **Flipkart and YouTube** use a mix: REST for external users and RPC for internal heavy lifting (inventory, processing).
