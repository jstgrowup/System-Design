## 📝 Detailed Notes

### Introduction to Consistency Models

- Consistency models are the **rules** that determine when a "Write" operation becomes visible to a "Read" operation in a distributed service.
- In microservices, you have multiple servers (nodes). If a user writes to Node A and a load balancer sends their next read request to Node B before they sync, a data gap occurs.

### Strong Consistency (Linearizability)

- **Definition:** Also known as **Strict Consistency** or **Linearizability**. It guarantees that all future reads will return the absolute latest value after a write happens.
- **The Sync Rule:** A "Write Success" status is only sent to the client after the data has been successfully synced across **all replicas**.
- **Mechanisms:** Uses leader-based architecture where the leader ensures all slaves have copied the data before acknowledging the write.
- ⚠️ **Warning:** This leads to higher latency because the system waits for full synchronization.
- 🎯 **Interview Tip:** Strong consistency is non-negotiable for **Banking and Financial transfers** where an incorrect balance could bankrupt a system.

### Eventual Consistency

- **Definition:** If no new writes occur, all replicas will **eventually** converge to the same value.
- **The Experience:** It allows for "Temporary Inconsistency." A user might see different data on different devices for a short window.
- **Replication Lag:** The delay in data copying caused by physical distance between nodes or network barriers.
- 💡 **Analogy:** LinkedIn likes. You might see 100 likes on your phone while your friend sees 101, but after a few seconds (once the background sync completes), you both see 102.

### Advanced Consistency Models

- **Sequential Consistency:** Operations appear in the same order on all nodes, but not necessarily in real-time.
- **Causal Consistency:** Ensures that if Event A causes Event B, everyone sees A before B.
  - 💡 **Example:** On social media, an "Edit" should never be visible before the "Original Post".
- **Read-Your-Writes (Read-Write Consistency):** Guarantees a user always sees their own updates immediately (often by storing the update locally first).
- **Monotonic Reads:** Once you see a specific version of data, you will never see an older version again (common in MongoDB).

### CAP & PACELC Theorems

- **CAP Recap:** You can only pick two: **C**onsistency, **A**vailability, or **P**artition Tolerance. In a distributed system, P is required, forcing a choice between C (Banking) or A (Social Media).
- **PACELC Extension:**
  - **P**artitioned: If there is a network partition (**P**), choose between Availability (**A**) or Consistency (**C**).
  - **E**lse (Normal operation): Choose between Latency (**L**) or Consistency (**C**).
- 💡 **Insight:** Even without failures, strong consistency (C) results in more latency (L), while eventual consistency offers less latency.

### ACID vs. BASE

- **ACID (SQL Focus):**
  - **Atomicity:** All-or-nothing transactions.
  - **Consistency:** DB follows all constraints (e.g., no negative bank balances).
  - **Isolation:** Concurrent transactions don't interfere with each other.
  - **Durability:** Data is safe even if the system crashes.
- **BASE (NoSQL Focus):**
  - **B**asically **A**vailable.
  - **S**oft State (Values may change even without new input).
  - **E**ventual Consistency.

## 📊 Diagrams & Visual Explanations

### 1. Strong Consistency Acknowledgement Flow

```text
[Client] --(1. Write)--> [Leader Node]
                             |
         -----------------------------------------
         | (2. Syncing...)    | (2. Syncing...)  | (2. Syncing...)
    [Replica 1]          [Replica 2]        [Replica 3]
         |                    |                  |
         ----(3. All Synced Acks)-----------------
                             |
[Client] <--(4. Write Success)---
```

_Description: Success is only returned after all replicas are updated, ensuring any subsequent read is accurate._

### 2. PACELC Decision Logic

```text
IF Partition (P)
   THEN choose (A)vailability OR (C)onsistency
ELSE (E)
   THEN choose (L)atency OR (C)onsistency
```

_Description: This extension explains the trade-offs during both failure (P) and normal operations (E)._

### 3. ACID Transaction State Machine

```text
[Active] --> [Partial Commit] --(Fail)--> [Rollback/Failed]
   |             |
(Fail)           --(Success)--> [Fully Committed] --> [Terminated]
```

_Description: Demonstrates the "All-or-Nothing" nature of Atomicity._

## 💻 Code Snippets & Commands

### ✅ Shown in video: Quorum Formula

```text
// To guarantee consistency in a system with N replicas:
Read_Quorum (R) + Write_Quorum (W) > Total_Replicas (N)
```

_If this condition is met, the system ensures that at least one node in a read operation has the latest write._

### 🧪 Generated learning example: ACID Transaction (SQL)

```sql
-- Conceptual example of Atomicity in banking
BEGIN TRANSACTION;
    UPDATE Accounts SET balance = balance - 100 WHERE id = 'User_A';
    UPDATE Accounts SET balance = balance + 100 WHERE id = 'User_B';
    -- If any step fails, the system triggers a ROLLBACK (All or nothing)
COMMIT;
```

## 🔁 Comparisons & Trade-offs

| Feature          | Strong Consistency      | Eventual Consistency      |
| :--------------- | :---------------------- | :------------------------ |
| **Visibility**   | Immediate               | Delayed (Eventual)        |
| **Latency**      | High (Waiting for sync) | Low (Background sync)     |
| **Availability** | Lower during partitions | Higher                    |
| **Use Case**     | Banking, Payments       | Social Media Likes, Feeds |

## 🎯 Interview Questions

1.  **What is Linearizability?** It is another term for Strong Consistency, ensuring all nodes see the latest write immediately.
2.  **Why can't we use Eventual Consistency for a banking ledger?** Because "Stale Reads" could allow a user to withdraw more money than they actually have if their balance isn't updated in real-time.
3.  **Explain Replication Lag.** It is the time difference between a write on the master and that write appearing on a replica, often caused by network distance.
4.  **What is the "Last Write Wins" (LWW) strategy?** A conflict resolution method in eventual consistency where the update with the latest timestamp is preserved.
5.  **How does PACELC improve upon the CAP theorem?** CAP only describes behavior during a network partition; PACELC adds the Latency vs. Consistency trade-off for when the system is running normally.

## ⚡ Quick Revision Flashcards

2.  **Q:** What does "Soft State" mean in BASE? **A:** The data may change over time even without new writes due to eventual sync.
3.  **Q:** What is the formula for Quorum consistency? **A:** $R + W > N$.
4.  **Q:** Which model is best for an E-commerce cart? **A:** Read-Your-Writes Consistency.
5.  **Q:** What is the trade-off for choosing Strong Consistency? **A:** Increased Latency.
6.  **Q:** What is a "Stale Read"? **A:** Reading old data that hasn't been updated yet.
7.  **Q:** Is MongoDB strong or eventual? **A:** It supports Monotonic Reads and can be configured for Strong/CP.
8.  **Q:** What is Causal Consistency? **A:** Ensuring related events (Post -> Edit) appear in order.
9.  **Q:** What is "Duration" in ACID? **A:** Data remains saved even if the system crashes.
10. **Q:** Where is BASE used? **A:** In NoSQL databases.

## 📌 Key Takeaways

- Consistency is a **spectrum** between speed (Eventual) and accuracy (Strong).
- **Banking** requires ACID/Strong Consistency; **Social Media** thrives on BASE/Eventual Consistency.
- **PACELC** is the modern standard for evaluating system trade-offs.
- To guarantee consistency in distributed nodes, use the **Quorum formula** ($R + W > N$).
- **Replication Lag** is an unavoidable reality of distributed systems due to physics and network barriers.
