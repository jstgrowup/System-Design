### Recap: What is System Design?

- It is the technical study of how a website or app works, including the choice of tech stacks.
- **Core Objectives:**
  - **Efficiency:** Ensuring the system works optimally.
  - **Handling Growth:** Managing massive traffic spikes (e.g., Flipkart Diwali sales or Big Billion Days).
  - **Reliability:** Making the system sustainable under pressure.
  - **Performance:** Ensuring high-speed execution.

### The Core Ecosystem: Front-end, Back-end, and Database

- **Front-end:** The visual interface users see (e.g., Instagram login screen, feed, or chat box).
- **Back-end:** The logic layer that validates user data.
- **Database (DB):** Where user info (passwords, chats) is stored.
- **The Flow:** Front-end sends a request $\rightarrow$ Back-end checks DB $\rightarrow$ If valid (True), it returns data (feed/chats); if invalid (False), it returns an error.
- 💡 **Analogy:** Instagram's chat box—you see the UI (Front-end), but the system ensures you only see your own chats, not someone else's (Back-end/DB logic).

### Database Paradigms: SQL vs. NoSQL

- **SQL:** Data stored in structured, tabular forms.
- **NoSQL:** Data stored in Key-Value pairs or JSON-like formats, allowing for flexible schemas.

### Optimization Techniques: CDN and Caching

- **CDN (Content Delivery Network):** Used for static data that remains constant for all users (e.g., Google.com’s home page) to avoid repeated back-end hits.
- **Caching:** Storing data locally to reduce expensive database queries.
- 🎯 **Interview Tip:** If asked what a cache is, explain that it saves database read time by storing frequently accessed data in a faster storage layer.
- ⚠️ **Warning:** Only cache data that doesn't change frequently; if a table has 1 crore (10 million) rows but only updates once a month, caching the result of a `SELECT *` query for 30 days is highly efficient.

### Traffic Management: Load Balancers

- **Load Balancer:** Distributes incoming requests across multiple servers or "pods" to prevent any single server from crashing.
- 💡 **Analogy:** Flipkart's Big Billion Days. If the system usually handles 1 lakh (100k) users but suddenly gets 1 crore (10 million) users (a 100x surge), a load balancer ensures the traffic is divided so the server doesn't "burst".

### High-Level Design (HLD) vs. Low-Level Design (LLD)

- **High-Level Design (HLD):** The "bird's-eye view." Focuses on components, API structures, scaling strategies, and overall architecture.
- **Low-Level Design (LLD):** The "deep dive." Focuses on class diagrams, Entity-Relationship (ER) diagrams, database schemas, and specific data attributes (e.g., a User object having a hashed password, bio, and profile pic).

### Step 1: Requirements and Flow Analysis

- Define the scope and scale (e.g., designing a URL shortener for 1 million users).
- Ask critical questions: How will it scale? Is it secure? Is it reliable if traffic jumps to 10 million?.

### Step 2: API Design (Endpoints and Data Contracts)

- Define the back-end endpoints first.
- Example: A `/feed` API needs inputs like `user_id` and a `limit` (to prevent the app from hanging by loading 1,000 posts at once).

### Step 3: Capacity Planning (Read/Write Throughput)

- Calculate **Requests Per Second (RPS)**.
- 💡 **Analogy:** Writing names in a register. A person can write maybe 10-100 entries a minute. A DB also has limits.
- If a DB supports 3,000 reads/sec and 1,000 writes/sec, but you receive 5,000 requests, you need multiple DB instances.

### Step 4: High-Level Architecture & Tech Stack (MERN)

- Choose the tech stack. The instructor highlights **MERN**:
  - **M**ongoDB (Database)
  - **E**xpress.js (Back-end framework)
  - **R**eact.js (Front-end)
  - **N**ode.js (Runtime environment).
- Hosting example: AWS.

### Step 5: Advanced Components (CDNs, Queues, and Scaling)

- **CDNs & Geography:** Data centers should be physically close to users. Data coming from Mumbai to Delhi is faster than data coming from the USA or Japan.
- **Queues:** If the back-end limit is 1,000 users but 2,000 arrive, use a queue to process the first 1,000 and hold the rest for the next cycle.
- 💡 **Analogy (Carry Minati Link):** If a famous creator like Carry Minati shares a link that gets 1 million clicks in a minute, that specific link should be moved to a Cache DB (like Redis) rather than querying the main DB every time.

### Step 6: Data Consistency, Security, and Hashing

- **Consistency:** Ensuring all platforms (web vs. mobile) show the same data.
- ⚠️ **Warning:** You might see a "bug" on LinkedIn where a "Like" takes time to reflect across devices; this is a synchronization/consistency lag.
- **Security:** Use hashing for passwords and implement safety features to prevent hacking.

### Step 7: Reliability and Reverse Engineering

- **Reverse Engineering:** Intentionally try to break your system to find failure points.
- **Monitoring & Alerts:** Set up systems (like on AWS) to send email alerts the moment a service fails so it can be fixed immediately.

## 🔁 Comparisons & Trade-offs

| Concept A    | Concept B         | When to use which                                                                                       |
| ------------ | ----------------- | ------------------------------------------------------------------------------------------------------- |
| **SQL**      | **NoSQL**         | Use SQL for structured, tabular data; use NoSQL for flexible, high-speed key-value storage.             |
| **HLD**      | **LLD**           | Use HLD for overall architecture/scaling; use LLD for detailed class and DB schema design.              |
| **Main DB**  | **Cache (Redis)** | Use Main DB for the source of truth; use Cache for frequently accessed, slow-changing data to save RPS. |
| **Read RPS** | **Write RPS**     | Systems usually have higher Read RPS; if Write RPS is too high, you need more DB instances.             |

## 🎯 Interview Questions

1. **What is the first thing you should do when given a system design problem?**
   Define the requirements, user flow, and the scale of the system (e.g., 1 million users).
2. **How does a Load Balancer help during traffic spikes?**
   It distributes the traffic across multiple server pods so no single instance is overwhelmed and "bursts".
3. **What is the difference between HLD and LLD?**
   HLD covers high-level components and APIs; LLD covers internal logic, classes, and DB schemas.
4. **Why would you use a CDN?**
   To store static data closer to the user's geographic location, reducing latency.
5. **How can you optimize a system that queries a 10-million-row table monthly?**
   Fetch the data once and store it in a cache for the duration of the month to avoid redundant heavy SQL queries.

## 📌 Key Takeaways

- System design is about making systems **efficient, reliable, and scalable**.
- Always clarify **requirements and scale** before building anything.
- **HLD** is for the big picture; **LLD** is for the code/data structure.
- **Caching** and **CDNs** are essential for reducing database load and user latency.
- Capacity planning (RPS) determines how many **database instances** you need.
- A robust system includes **monitoring and automated alerts** for failure recovery.
