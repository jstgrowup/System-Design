## 📝 Detailed Notes

### Introduction: Thinking Beyond the Code

- System design is about how a solution solves people's problems, not just the code written.
- **Primary Goal:** Solve the problem while optimizing for the **minimum budget** and maximum efficiency.

### The System Designer’s Mindset

- **Coder vs. System Designer:** A coder thinks about how the feature works; a designer thinks about how the system handles **1 million requests** simultaneously.
- Focus on the "Whole System Flow" rather than the implementation details of a single function.
- ⚠️ **Warning:** During the design phase, do not think like a coder. You are a designer; coding comes much later.

### Interview Strategy: Communication is Key

- In an interview, you must **talk through your thoughts** more than you write.
- Explain what you understand about the problem before jumping into technical stack choices.

### Core Design Questions: Problem, Scale, and Price

- Before picking a tech stack, ask:
  - What problem am I solving?
  - At what **scale** (users/traffic) am I solving it?
  - What is the **cost/price** of this solution?
- 💡 **Analogy:** Early computers took a long time just to do $1 + 1$. Today’s systems handle millions of actions per second because of design, not just faster processors.

### Case Study: Designing YouTube

- **Step 1: Identify Users:** Who are they? Where are they located (geographical location)?
- **Step 2: Main Goals:**
  - Uploading videos.
  - Viewing videos.
  - Sharing videos.
- **Step 3: Side/Secondary Goals:** Likes, comments, and other engagement features.

### Requirement Gathering

#### Functional Requirements (What it does)

- Users can upload and view videos.
- Users can create channels.
- Users can subscribe to channels and get notifications.
- Profile management.

#### Non-Functional Requirements (How it performs)

- **Scalability:** Handle massive concurrent traffic.
- **Low Latency:** Videos should load/process quickly. Processing different pixels (360p, 1080p) shouldn't take 5 hours.
- **Reliability:** If a video is uploaded, it shouldn't disappear a year later.
- **Availability:** The system must be up 24/7 (even at 3 AM).
- **Consistency:** If a video is edited, the change should reflect immediately for users.

### The Hierarchy of Design

1.  **High-Level Design (HLD):** Identifying major components like API Gateways, Load Balancers, DB types, CDNs, Queues, and Storage.
2.  **Data Flow:** Mapping how data moves. For YouTube: Video ID → Link to storage → Views/Likes/Comments tracking.
3.  **Low-Level Design (LLD):** Deciding on specific tools (e.g., **S3 Bucket** for storage, **RabbitMQ** for queues) and defining API endpoints (e.g., `/upload` body and response).

### Optimization: Caching, Scaling, and Fault Tolerance

- **Caching:** Use caching to reduce the load on the main Database.
- **Auto-Scaling:** Use platforms like **AWS** to scale the system automatically based on demand.
- **SQL vs. NoSQL:** Choose based on the specific needs of the data (structured vs. flexible).
- **Fault Tolerance:** If one component fails, the whole system shouldn't go down. It should recover quickly without data loss.

### Security and Privacy

- Ensure the system is secure. If a user marks a video as **Private**, the system must ensure it is not visible publicly.

### The Conclusion: No Strict Rulebook

- There is no single "correct" way to design. The best design is the one that meets the specific requirements of your users and budget.
- 🎯 **Interview Tip:** Use AI or docs (like ChatGPT) to research best practices for specific components during your preparation, but remember that the designer’s logic is what matters most in the room.

## 🎯 Interview Questions

1. **What is the difference between a coder and a system designer?**
   A coder focuses on feature logic, while a designer focuses on how that logic scales, its reliability, and its cost-efficiency.
2. **What are the non-functional requirements for a video platform like YouTube?**
   Scalability, low latency (fast processing), reliability (no data loss), 24/7 availability, and consistency across devices.
3. **Why is "Price" or "Budget" a consideration in system design?**
   A good designer must solve problems with a _minimum budget_, ensuring the company doesn't overspend on resources while maintaining performance.
4. **How do you handle "Fault Tolerance" in a system?**
   By ensuring that if one part fails, the system recovers automatically and doesn't delete or lose user data.
5. **What is the first step when designing a system like YouTube?**
   Identifying the users, their geographical locations, and the core functional goals (upload/view/share).

## 📌 Key Takeaways

- System design is **problem-solving**, not just coding.
- Always distinguish between **Functional** (features) and **Non-Functional** (performance) requirements.
- **Scale and Cost** are the two biggest constraints a designer faces.
- A structured approach moves from **Users → Goals → HLD → Data Flow → LLD**.
- **Reliability and Consistency** ensure the user experience remains stable even during edits or crashes.
