## 📝 Detailed Notes

### Introduction & Core Concept

- **Context:** This is the 8th episode of the series, moving from fundamentals like Latency and HTTP to core infrastructure components.
- **The Problem:** In a standard client-server model, if a video goes viral (e.g., on Instagram or YouTube), 10 million users might request the same resource simultaneously.
- **Server Limits:** Every server has a capacity limit (RAM, CPU). If a server can handle 1 million requests but receives 10 million, it will crash ("burst") and stop working entirely.
- **The Solution:** Use multiple server replicas (e.g., 10–12 machines) and a **Load Balancer** to divide the load among them.

### Definition & Benefits

- **Definition:** A Load Balancer sits between the client and the server as an intermediary "wall" or "distributor".
- 💡 **Analogy:** It acts like a **Traffic Police** officer directing cars to different lanes to prevent congestion.
- **Core Benefits:**
  - **Availability:** If one server goes down, traffic is redirected to remaining healthy servers.
  - **Scalability:** Allows for "Auto-scaling," where new servers are added automatically when traffic hits a threshold (e.g., 90% load).
  - **Reliability:** Ensures a backup is always ready if a primary server fails.
- 💡 **Analogy:** A mall with only one billing counter creates long lines. Adding 2–3 more counters allows customers to be served quickly and prevents the system from stalling.

### Types of Load Balancers

1.  **Hardware Load Balancers:**
    - Physical devices (e.g., Citrix, F5) that sit between optical wires.
    - **Pros:** Extremely fast and very safe for data.
    - **Cons:** Very expensive (costing millions), inflexible, and mostly used by large enterprises and banks.
2.  **Software Load Balancers:**
    - Applications installed on servers (e.g., **Nginx**, **HAProxy**).
    - **Pros:** Cheap, flexible, popular, and easy to use.
3.  **Cloud Load Balancers:**
    - Provided by platforms like **AWS (Elastic Load Balancer - ELB)**, Google Cloud (GCP), or Azure.
    - **Pros:** High availability, managed by the provider, and includes built-in auto-scaling features.

### Load Balancing Algorithms (Crucial for Interviews)

- **Round Robin:** Requests are sent in a circular, sequential order (1, 2, 3, 1, 2, 3...). It doesn't consider server load or user identity.
- **Sticky Round Robin:** Ensures all subsequent requests from a specific user go to the same server they initially connected to.
- **Weighted Round Robin:** Servers are assigned "weights" based on their power (e.g., a server with 10x more RAM gets a weight of 8, receiving more traffic).
- **IP/URL Hash:** Hashes the user's IP or URL to a specific number; users with the same hash always go to the same service.
- **Least Connection:** Traffic is directed to the server with the lowest number of active connections/load.
- **Least Time:** Traffic is directed to the server providing the fastest response time.
- **Random:** Requests are distributed randomly. Statistically, load eventually splits roughly equally (e.g., 1/3 each for 3 servers).

### Layer 4 vs. Layer 7 Load Balancing

- **Layer 4 (Transport Layer):**
  - Works on IP and Port numbers; often called TCP/UDP load balancing.
  - **Pros:** Extremely fast, high performance, and no request inspection.
  - **Cons:** No "smart routing" because it doesn't look at the data inside the request.
- **Layer 7 (Application Layer):**
  - Works on HTTP, URLs, and Cookies.
  - **Pros:** Enables **Intelligent Routing** (e.g., `/api` requests go to Server A, while `/images` go to Server B).
  - **Cons:** Slower than Layer 4 because it must inspect the content, but more flexible and cheaper to implement.

### Health Checks & Session Persistence

- **Health Checks:** The Load Balancer continuously "pings" backend servers (Ping/Pong). If a server fails to respond, it is removed from the rotation, and a new instance is launched via auto-scaling.
- **Session Persistence:** Essential when a user logs into Server A but their next request goes to Server B (where they aren't logged in).
  - **Solutions:**
    - **Sticky Sessions:** Use cookie-based logic to keep the user on one server.
    - **External Store (Redis):** Store session data in a central Redis cache. Any server can check Redis to verify the user is logged in.
    - **JWT (JSON Web Tokens):** A stateful/stateless token approach for authentication.

### Advanced Strategy: Avoiding SPOF

- **Single Point of Failure (SPOF):** If you only have one Load Balancer and it fails, the entire system crashes.
- **The Fix:** Deploy **Multiple Load Balancers** (e.g., Active-Passive or Active-Active configurations) so that if one fails, others take over.

### Load Balancer vs. API Gateway

- **Load Balancer:** Primarily focused on **Traffic Distribution**.
- **API Gateway:** Focused on **API Management**, including Authentication, Rate Limiting, and complex Hierarchical Routing. It usually sits _behind_ the Load Balancer.

### 🧪 Generated learning example: Round Robin Logic

```javascript
// A simple conceptual implementation of Round Robin
const servers = ["Server_A", "Server_B", "Server_C"];
let currentIndex = 0;

function getNextServer() {
  const server = servers[currentIndex];
  currentIndex = (currentIndex + 1) % servers.length; // Circular order
  return server;
}

console.log(getNextServer()); // Server_A
console.log(getNextServer()); // Server_B
console.log(getNextServer()); // Server_C
console.log(getNextServer()); // Server_A (starts over)
```

## 🔁 Comparisons & Trade-offs

## 🎯 Interview Questions

1.  **Can a Load Balancer be a Single Point of Failure?**
    Yes, if only one exists. Solution: Use multiple Load Balancers (Active-Passive).
2.  **What is the difference between Least Connection and Least Time algorithms?**
    Least Connection targets the server with the fewest active tasks; Least Time targets the one with the fastest response speed.
3.  **Explain the "Sticky Session" problem and how to solve it.**
    Users might lose login state if moved to a different server. Solutions include Sticky Round Robin, central Redis storage, or JWTs.
4.  **Why are Hardware Load Balancers used by banks?**
    Because they are physical devices that offer high speed and extreme security for wired communications.
5.  **What happens if a server fails a health check?**
    The Load Balancer stops sending traffic to it and redirects requests to remaining healthy servers while auto-scaling may launch a replacement.

## 📌 Key Takeaways

- Load Balancers prevent **server crashes** by distributing traffic.
- **Horizontal Scaling** relies heavily on Load Balancers.
- **Software Load Balancers** (Nginx) are preferred for most web apps due to cost and flexibility.
- **Round Robin** is the simplest algorithm, but **Least Connection** and **Least Time** are often more efficient.
- Always design with **redundancy** (Multiple LBs) to avoid total system failure.
