## 📝 Detailed Notes

### Monolithic Architecture: Definition and Core Structure

- **Etymology:** "Mono" means one, and "Lithic" means structure; it refers to an application built as a single, unified unit.
- **Structure:** All code (Front-end, Back-end, Database logic, Auth, Payments) resides in a single repository and runs on a single machine or server.
- **Internal Communication:** Since everything is in one place, internal communication between modules is fast.
- **Scaling:** Primarily requires **Vertical Scaling** (adding more RAM/CPU to the single machine), though it can be horizontally scaled to an extent by duplicating the entire monolith.
- 💡 **Analogy:** It is like a single large folder containing sub-folders for every feature; if you want to deploy a change, the entire folder must be moved together.

### Tools and Real-world Examples

- **Frameworks:** Popular frameworks that support monolithic builds include **Ruby on Rails**, **Django** (Python), and **Next.js** (JavaScript).
- **Initial Versions:** Many giants like **Instagram**, **Amazon**, and **Shopify** started as monoliths.
- **Case Study:** **Coding Ninjas** uses a single repository built with Ruby on Rails, where all code for different features is stored together.

### Pros and Cons of Monoliths

- **Advantages:**
  - **Easier Development:** No need to manage complex API links or request-response cycles between services initially.
  - **No CORS Issues:** Since it's one origin, Cross-Origin Resource Sharing setup isn't required.
  - **Simplicity:** Easier deployment (one place), easier debugging, and lower infrastructure costs.
- **Disadvantages:**
  - **Scaling Limits:** Eventually, vertical scaling hits a physical limit.
  - **Conflict Heavy:** Large teams (e.g., 100+ developers) will face constant merge conflicts when working in the same repo.
  - **Technology Lock-in:** You are stuck with one language/framework (e.g., if it's RoR, every developer must know Ruby).

### Microservices Architecture: Decentralized Services

- **Definition:** Small, independent services running on separate machines, each handling a specific task (e.g., Payment service, Auth service, User Management).
- **Communication:** Services interact via a "web" of requests and responses.
- **Independent Scaling:** You can scale only the service under heavy load (e.g., scaling the "Order" service during an Amazon sale without scaling the "Notification" service).

### The API Gateway

- **Central Entry Point:** Clients (Mobile, Web) never communicate directly with internal services; they go through the Gateway.
- **Responsibilities:** Routing requests to the correct service, Authentication, Rate Limiting, Logging, and Load Balancing.
- **Tools:** **Kong** (very popular and easy), **Nginx**, **AWS API Gateway**, and **Apigee**.

### Service Discovery & Registry

- **The Problem:** In microservices, instances (pods) are dynamic—they start, stop, and change IP addresses constantly.
- **Service Registry:** A central database that stores the current location and IP of all active services.
- **Service Discovery:** The automated process where the API Gateway or Load Balancer queries the Registry to find where to route a request (e.g., asking "What is the current IP for the Payment service?").

### Synchronous vs. Asynchronous Communication

- **Synchronous (Sync):** The process stops and waits for a response (e.g., adding to basket $\rightarrow$ checking catalog $\rightarrow$ ordering).
- **Asynchronous (Async):** The process doesn't stop; it puts a task in a **Queue** or **Message Broker** and continues (e.g., signing up a user and immediately showing the profile while the Welcome Email is handled in the background).
- ⚠️ **Warning:** Synchronous chains are "Anti-patterns" if they make the user wait too long; if one step fails in a sync chain, the whole process fails.

### Fault Isolation & Circuit Breaker

- **Fault Isolation:** If the Payment service fails, the Home page and User profile services continue to work.
- **Circuit Breaker Pattern:** If a service (like Payments) fails repeatedly, the **Service Broker** temporarily stops sending requests to it. This prevents "cascading failures" and allows the failing service time to refresh and regain power without being overwhelmed by a backlog of requests.

## 📊 Diagrams & Visual Explanations

### 1. Monolithic Structure

```text
[   Unified Application Server   ]
|-- [ Auth Module ]              |
|-- [ Payment Module ]           | ---> [ Single Database ]
|-- [ Order Module ]             |
|-- [ Notification Module ]      |
```

_Description: Everything runs in one single repo and machine, sharing one database._

### 2. Microservices with API Gateway

```text
[ Client ] --> [ API GATEWAY ]
                    |
      ------------------------------
      |             |              |
[ Auth Service ] [ Order Service ] [ Payment Service ]
      |             |              |
  [ DB 1 ]       [ DB 2 ]       [ DB 3 ]
```

_Description: Requests hit the Gateway first, which routes them to independent services that may have their own databases._

### 3. Service Discovery Flow

```text
1. [Service] --(Register IP)--> [Service Registry]
                                      ^
2. [API Gateway] --(Where is Payment?)--|
                                      |
3. [API Gateway] <--(Here is the IP)---|
```

_Description: The Registry acts as a phonebook for dynamic service IPs._

## 🔁 Comparisons & Trade-offs

| Feature        | Monolithic Architecture           | Microservices Architecture              |
| :------------- | :-------------------------------- | :-------------------------------------- |
| **Codebase**   | Single Repo (Unified)             | Independent Repos per service           |
| **Deployment** | Single Deployment                 | Independent Deployments                 |
| **Scaling**    | Scale the whole system            | Scale individual services               |
| **Technology** | Technology Lock-in (Single stack) | Technology Flexibility (Polyglot)       |
| **Complexity** | Low (Initially)                   | High (Requires management)              |
| **Faults**     | One failure can stop everything   | Fault isolation (One fails, others run) |

## 🎯 Interview Questions

1. **What does the "Lithic" in Monolithic mean?**
   It refers to a structure; "Monolithic" implies a single, unified structure for the entire application.
2. **Why do large companies move away from Monoliths?**
   Because large teams face merge conflicts, technology lock-in, and the inability to scale specific components independently.
3. **What is the primary responsibility of an API Gateway?**
   It acts as a central entry point for routing, authentication, rate limiting, and load balancing.
4. **How does Service Discovery handle dynamic IPs?**
   A Service Registry stores all current service locations, and the Gateway queries it to find the correct IP before routing.
5. **Explain the Circuit Breaker pattern.**
   It prevents cascading failures by temporarily stopping requests to a service that is repeatedly failing, allowing it time to recover.
6. **When should you choose a Monolith over Microservices?**
   When you have a small team, a small product, need fast development, and have low initial traffic.

## 📌 Key Takeaways

- **Monoliths** are best for **speed and small teams**; **Microservices** are for **complexity and massive scale**.
- The **API Gateway** is the "security guard" and "receptionist" for your microservices.
- **Service Discovery** is required because service IPs change constantly in the cloud.
- **Async communication** (Queues) is vital for a smooth user experience in microservices.
- Use a **Circuit Breaker** to stop a small failure from becoming a total system blackout.
