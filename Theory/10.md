## 📝 Detailed Notes

### Introduction to SQL (RDBMS)

- **SQL (Structured Query Language):** The standard language for managing **Relational Database Management Systems (RDBMS)**.
- **Data Structure:** Data is stored in **tables** consisting of **rows** and **columns**.
  - **Columns (Attributes):** Define what is being stored (e.g., Name, Mobile, Email).
  - **Rows:** Contain the actual data values (e.g., "Shivansh", "999...", "email@...").

### Popular SQL Tools & Interfaces

- **Major Tools:** MySQL and PostgreSQL are the most popular; others include Oracle and Microsoft SQL Server.
- **Interfaces:**
  - **GUI (Graphical User Interface):** Tools like MySQL Workbench allow for visual data management and ER diagram creation.
  - **CLI (Command Line Interface):** Used for writing and executing direct queries.

### ER Diagrams (Entity-Relationship)

- **Definition:** A visual representation of how different entities (tables) connect to one another.
- 💡 **Example:** A `Purchase` table relates to a `Product` table (via Product ID) and a `Customer` table (via Customer ID).

### Key Properties of SQL

- **Fixed Schema:** You must define the structure (columns) beforehand. Every row must follow this structure, even if some values are NULL.
- **Strong Consistency:** Ensures data is uniform across the system.
- **Vertical Scaling:** Increasing capacity by adding more power (storage/RAM) to a single server.
- **ACID Compliance:** Ensures reliable transaction processing.

### Introduction to NoSQL (Not Only SQL)

- **Definition:** A **schema-less** database where data is stored in flexible formats like key-value pairs or documents.
- **Horizontal Scaling:** Scaling by adding more machines to a cluster, allowing for high write throughput.

### Types of NoSQL Databases

- **Document:** Stores data as objects (e.g., **MongoDB**).
- **Key-Value:** Simple pairs (e.g., **Redis**).
- **Wide Column:** Optimized for large datasets (e.g., **Cassandra**).
- **Graph:** Manages complex relationships (e.g., **Neo4j**).

### CAP Theorem: The Core of Distributed Systems

- **C (Consistency):** Every node in the system returns the same, most recent data at the same time.
- **A (Availability):** Every request receives a response, even if the data is slightly outdated (stale).
- **P (Partition Tolerance):** The system continues to function despite network failures or communication breaks between nodes.
- ⚠️ **The Rule:** You can only guarantee **two** of these three properties at once; achieving all three is impossible.

### CP vs. AP Strategies

- **CP (Consistency + Partition Tolerance):** Sacrifices availability. Used in **Banking and Payments** where showing an incorrect balance is unacceptable. The system may "wait" or show an error rather than provide wrong data.
- **AP (Availability + Partition Tolerance):** Sacrifices consistency. Used in **Social Media (Instagram/LinkedIn)**. It’s okay if "likes" show differently on a phone vs. a laptop for a few minutes as long as the app remains available.

### 2. CAP Theorem Triangle

```text
          Consistency (C)
             /      \
            /   [P]  \
    (CP)   /    is    \   (CA)
          /  Required  \
         /____________  \
Availability (A) ---- Partition Tolerance (P)
               (AP)
```

_Description: In a distributed system, Partition Tolerance (P) is mandatory, forcing a choice between CP and AP._

## 💻 Code Snippets & Commands

### ✅ Shown in video: SQL Query Examples

```sql
-- Creating a table with constraints
CREATE TABLE Users (
    id INT PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    age INT NOT NULL UNIQUE
);

-- Selecting all data with a condition
SELECT * FROM Users WHERE age > 18;

-- Inserting data
```

## 🔁 Comparisons & Trade-offs

| Feature       | SQL (Relational)                | NoSQL (Non-Relational)         |
| :------------ | :------------------------------ | :----------------------------- |
| **Schema**    | **Fixed** (Predefined)          | **Flexible** (Schema-less)     |
| **Scaling**   | **Vertical** (Bigger machine)   | **Horizontal** (More machines) |
| **Joins**     | **Supported** (Complex queries) | **Rare** (Usually avoided)     |
| **Integrity** | ACID Compliant (High)           | CAP/BASE focused               |
| **Data Type** | Structured                      | Unstructured/Semi-structured   |

## 🎯 Interview Questions

1. **When would you choose SQL over NoSQL?**
   When the system requires a fixed schema, strong data integrity, and complex joins, such as in banking or inventory management.
2. **What is the significance of 'P' in the CAP theorem?**
   Partition Tolerance ensures the system survives network failures. In distributed systems, P is usually non-negotiable.
3. **Why do Social Media apps prefer AP over CP?**
   Because availability is more important than immediate consistency. Users would rather see a post with a slightly wrong like count than have the app crash or freeze.
4. **How does horizontal scaling benefit NoSQL?**
   It allows for high write throughput by distributing data across many machines, making it ideal for big data and real-time analytics.
5. **What is an ER Diagram?**
   An Entity-Relationship diagram is used to visualize how different tables and their attributes are interconnected in a database.

## 📌 Key Takeaways

- **SQL** = Structured, Fixed, Consistent, Vertical.
- **NoSQL** = Flexible, Schema-less, Scalable, Horizontal.
- **CAP Theorem:** You can only pick two (C+P or A+P in distributed systems).
- **Banking** needs **CP** (Consistency); **Social Media** needs **AP** (Availability).
- Always select your database based on the **specific features** (e.g., SQL for profiles, NoSQL for feeds).
