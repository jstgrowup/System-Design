## 📝 Detailed Notes

### 1. Role of System Design in Careers

- System design concepts appear in basic interview questions and are vital for companies like Optum, Microsoft, Google, etc.
- 🎯 **Interview Tip:** Sharing real-world project experiences (like work at Coding Ninjas or Optum) helps demonstrate practical knowledge.

### 2. Recap: Core Purpose of System Design

- It defines how a website or app works technically (the "Tech Stack").
- Focuses on making a system **efficient**, handling **growth** (e.g., Flipkart Big Billion Days), and ensuring **reliability** and **performance**.

### 3. Front-end vs. Back-end (The Interaction Flow)

- **Front-end:** The visual screens (Login, Feed, Chat box with sender/receiver UI).
- **Back-end:** The logic layer that validates credentials against the Database (DB). If the password matches, it returns the feed/chats; otherwise, it returns an error.
- 💡 **Analogy:** Instagram's login—if you enter the right password, the backend fetches _your_ specific data, not someone else's.

### 4. Database Basics: SQL vs. NoSQL

- **SQL:** Data stored in a structured, **tabular form**.
- **NoSQL:** Data stored in **key-value pairs** or JSON format (flexible for varying data).

### 5. Optimization: CDNs, Caching, & Load Balancers

- **CDN (Content Delivery Network):** Stores static data (like the Google home page) so it doesn't hit the backend every time.
- **Caching:** Storing data locally or in a fast layer (like **Redis**) to avoid expensive DB queries.
  - 💡 **Example:** If a table with 1 crore (10 million) records only updates once a month, don't run `SELECT *` every time. Cache the result for that month.
- **Load Balancer:** Manages traffic spikes. If daily traffic is 1 lakh but hits 1 crore during a sale, the load balancer divides requests among multiple "Pods" (service instances) to prevent the server from "bursting".

### 6. HLD vs. LLD Review

- **LLD (Low-Level Design):** Focuses on class diagrams, DB schemas, and ER diagrams (e.g., a User schema with hashed passwords, bio, and profile pic).
- **HLD (High-Level Design):** The "big picture" of how APIs, scaling, and components like "Feed" or "Auth" connect.

### 7. The Interview Flow: Step-by-Step Approach

To design a system (like a URL Shortener) in an interview, follow this sequence:

1.  **Requirement Gathering:** Define the scale (e.g., 1 million users) and functional needs.
2.  **API Design:** Define endpoints (e.g., `/feed`).
    - ⚠️ **Warning:** Always set **limits** on API outputs (e.g., fetch 10 posts, not 1000) or the app will hang.
3.  **Capacity Planning:** Calculate Read vs. Write operations per second (RPS).
4.  **High-Level Architecture:** Select tech stack (e.g., MERN, AWS, SQL).
5.  **Scalability & Optimization:** Add CDNs, Caching, and Queues.
6.  **Security & Consistency:** Implement hashing and handle data lag (e.g., the LinkedIn "Like" bug).
7.  **Reverse Engineering:** Identify failure points and set up monitoring/alerts.

### 8. Capacity Planning (RPS Math)

- Systems have different limits for **Read** and **Write** operations.
- 💡 **Analogy:** Like a person writing in a register; you can only write so fast. A DB might handle 3000 reads but only 1000 writes per second. If 5000 write requests hit, you need more DB instances.

### 9. CDNs & Global Strategy

- 💡 **Example:** If your data center is in the USA but the user is in Mumbai, fetching a 10MB image is slow. A CDN keeps the data in a Mumbai data center for faster access.

### 10. Messaging Queues

- Used when requests exceed the server's immediate processing capacity.
- **Process:** If capacity is 1000 but 2000 requests arrive, 1000 go into a **Queue** for the next cycle.

## 🔁 Comparisons & Trade-offs

| SQL                                   | NoSQL                                |
| ------------------------------------- | ------------------------------------ |
| Tabular/Structured format.            | Key-Value/JSON format.               |
| Best for consistent, relational data. | Best for flexible, high-volume data. |

| Read Operation                        | Write Operation                        |
| ------------------------------------- | -------------------------------------- |
| Fetching data (e.g., viewing a feed). | Saving data (e.g., posting a comment). |
| Typically faster/higher capacity.     | Typically slower/lower capacity.       |

## 🎯 Interview Questions from This Video

1. **How would you handle a 100x traffic spike on a platform like Flipkart?**
   - _Answer:_ Use a **Load Balancer** to distribute traffic across multiple server pods and implement **Scalability** to add more instances automatically.
2. **What is the "LinkedIn Like Bug" and what causes it?**
   - _Answer:_ It's a **consistency** issue where a "like" reflects on mobile but takes time to show on the web because the data hasn't fully synchronized across all DB instances.
3. **When should you use Caching (like Redis)?**
   - _Answer:_ Use it for data that is frequently read but rarely changed, or for viral content (like a CarryMinati link) to avoid hitting the main DB repeatedly.
4. **Why are API limits important in System Design?**
   - _Answer:_ Without limits (pagination), a request might try to fetch too much data at once, causing the backend or front-end app to hang or crash.
5. **What is the role of a Queue in a high-traffic system?**
   - _Answer:_ A Queue acts as a buffer, holding requests that exceed the server's immediate processing limit to be handled in subsequent cycles.

## 📌 Key Takeaways

- System Design for interviews follows a **strict flow**: Requirements ➔ APIs ➔ Capacity ➔ Architecture ➔ Scaling.
- **Real-world examples** (Instagram, LinkedIn, CarryMinati) are essential for explaining concepts to interviewers.
- Always plan for **system failure** through reverse engineering and monitoring.
- **Performance vs. Consistency** is a constant trade-off (e.g., the LinkedIn bug).
- Optimization tools like **CDNs and Caching** are the "secret sauce" for handling global scale.
